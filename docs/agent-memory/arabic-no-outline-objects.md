For **Arabic** calligraphy, always add **`--no-outline-objects`** to the recipe.
`--outline-objects` is AUTO-on for satin-dominant categories (incl. arabic), and on
script it lays a **satin border around each core object** — the user calls this out as
wrong: "you did the first letter right, then did outline satin and fill ... its hard
requirement to follow production-ready work." Production Arabic is one clean object per
stroke, never a border-wrapped core.

**Why:** the border hybrid is double structure production never uses, and it inflated
density (1.1 → 0.84 st/mm², back into the reference 0.4–0.8 band) once removed.

**How to apply:** the faithful Arabic recipe (post-arb-trio, 2026-07-17) is
`--purify-colors --no-outline-objects` (+ chart) — **no fill-method flag, no
--satin-lean**: the `satin_only` pair prior drives the all-satin path automatically.
See [[arabic-satin-only-law]].

**SUPERSEDED paragraph (kept for history):** this memory used to warn "do NOT fix the
satin gap by forcing full satin — coverage collapsed 90 % → 70 % on tawfiq". That
tested the OLD `--satin-lean --vwidth-satin` path without the arb-trio machinery. The
satin-only law's version (measured width-band clamps, hairpin-split columns,
turning-satin rings, single-strip dots, authored spacing) converges: satin_frac 100,
fill_frac 0, all arcs complete on the arb rebuild. The honest limit that remains is
per-stroke granularity (merged columns are fatter than the master's), not coverage.
